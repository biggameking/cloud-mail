# Game Retrospective

## Trigger

Run after a prototype decision, milestone, external playtest wave, release, incident, or repeated workflow failure.

## Inputs

- Stage goals and exit gates.
- Build, playtest, quality, performance, content, schedule, and release evidence.
- Decision log, risk register, defects, rework, and prior retrospective actions.

## Responsibility Hats

- **Solo:** review separately as Creative, Production, Engineering, Content, and QA; delay synthesis until each pass is written.
- **Team:** Production facilitates; each discipline brings evidence; Product/Creative owns scope decisions; action owners accept due points.

## Steps

1. Restate intended outcomes and compare them with evidence, not effort.
2. Identify what improved player value, retired risk, or increased sustainable throughput.
3. Identify misses, rework, escaped defects, unclear decisions, content bottlenecks, and budget variance.
4. Trace each important result to controllable system causes; avoid blame and vague labels.
5. Review prior actions and close, continue, or replace them.
6. Choose at most a few high-leverage changes; assign owner, evidence, and review milestone.
7. Update canonical facts, stage gates, risk register, estimates, templates, or project-local devrules memory as appropriate.
8. Define the next stage's smallest player-visible outcome and highest risk.

## Outputs

- Evidence-based outcome review.
- Continue/stop/change decisions.
- Small action set with owners and verification points.
- Updated canonical sources and next-stage focus.

## Gates

- Claims reference observable evidence.
- Actions address causes within project control.
- Each action is bounded, owned, and testable.
- Process changes do not create more ceremony than the risk warrants.
- Durable project learning is captured in the proper local memory or canonical document.

## Done Criteria

Stage outcomes are accepted or rejected against their gates, improvement actions have owners and evidence dates, canonical plans reflect learning, and the next stage starts with reduced ambiguity and a smaller prioritized risk set.
